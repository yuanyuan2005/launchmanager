

# 软件工程项目计划书

---

# 第一部分：问题定义

---

### 1.1 问题背景

用户日常使用 Windows 系统时，桌面逐渐演变为一个混乱的存储空间：

-   **快捷方式堆积**：各类软件的快捷方式（`.lnk` 文件）散落在桌面，数量可达数十个，查找特定程序时需要肉眼扫描。
-   **文件混杂**：临时文档、下载的图片、压缩包等与快捷方式混合堆放，边界模糊。
-   **缺乏分类体系**：Windows 桌面仅提供“按名称/日期/类型排序”或“图标对齐网格”，无法进行自定义分类或分组。
-   **现有工具不理想**：
    -   商业软件（如 Fences）虽可分区管理，但付费且有广告捆绑。
    -   开源启动器（如 Flow Launcher、Wox）多为键盘流搜索工具，缺少可视化的图形分类管理。
    -   部分开源工具界面粗糙、年久失修，不符合用户“干净、简单”的审美需求。

### 1.2 问题陈述

**核心问题：** 用户缺乏一个**可视化、可自定义分类、一键呼出**的个人常用文件与程序管理工具。

**症状：**
1.  桌面混乱，查找效率低。
2.  常用程序与文件分散在不同磁盘位置，无统一入口。
3.  用户被迫在“忍受桌面混乱”和“使用带广告的商业软件”之间做选择。

### 1.3 用户期望

用户（即开发者本人）希望通过一个自研工具实现以下目标：

1.  **告别桌面图标**：将所有常用入口收纳进一个干净的面板，桌面本身可永远保持空白。
2.  **灵活分类**：通过“分组”和“标签”两个维度组织所有项目。
3.  **快速访问**：按全局热键呼出面板，搜索或浏览后一键启动，面板自动收回。
4.  **干净健康**：无广告、无捆绑、不联网、开源、绿色版。

### 1.4 问题定义结论

开发一个基于 Electron 的单机桌面应用，名为 **LaunchHub**，作为个人常用文件与程序的**总控制台**，替代混乱的 Windows 桌面。

# 第二部分：可行性研究

---

### 2.1 技术可行性

| 技术点 | 实现方案 | 可行性 | 依据 |
|-------|---------|--------|------|
| 桌面应用框架 | Electron | 高 | 开发者已有完整 Electron 项目经验，有可复用的基础框架 |
| 读取桌面文件 | Node.js `fs.readdir` | 高 | 标准库，无兼容性问题 |
| 解析 `.lnk` 快捷方式 | `shell.readShortcutLink()` (Electron 内置) | 高 | Electron 原生 API，无需第三方库 |
| 启动程序/打开文件 | `shell.openPath()` (Electron 内置) | 高 | 已有开发经验，API 稳定 |
| 数据持久化 | 本地 JSON 文件 | 高 | 技术成熟，适合单用户、小数据量场景 |
| 全局热键 | `globalShortcut` (Electron 内置) | 高 | Electron 原生支持 |
| 图形界面 | HTML/CSS/JS | 高 | 开发者最擅长的技术栈 |
| 打包分发 | electron-builder | 高 | 已有使用经验 |

**技术结论：** 所有技术点均有成熟方案或可直接复用既有经验，无技术障碍。

### 2.2 经济可行性

-   **开发成本：** 零。个人利用业余时间开发，AI 辅助编码，所有工具链（Node.js, Electron, VS Code）均为开源免费。
-   **维护成本：** 极低。仅需个人偶尔优化，无外部用户支持负担。
-   **收益：** 无直接经济收益。项目定位为个人效率工具，不计划商业化。

**经济结论：** 零预算，完全可行。

### 2.3 操作可行性

-   **用户角色：** 唯一用户即为开发者本人。
-   **学习成本：** 零。操作习惯完全由自己定义。
-   **文档需求：** 无需面向外部用户的操作手册。
-   **运行环境：** 仅需 Windows 10/11 系统。

**操作结论：** 操作阻力为零。

### 2.4 法律可行性

-   **数据范围：** 仅读取用户自行创建的快捷方式和文件路径，不触碰第三方软件本体。
-   **开源协议：** 项目代码完全自研，计划以 MIT 协议开源。
-   **不联网：** 不连接任何服务器，不收集任何数据。

**法律结论：** 无侵权风险，无隐私合规负担。

### 2.5 可行性研究结论

项目在技术、经济、操作、法律四个维度均高度可行，建议立即进入需求分析阶段。

# 第三部分：需求分析

---

### 3.1 用户场景

**场景一：日常工作启动**
> 小王到公司，按下 `Alt+空格`，LaunchHub 面板弹出。他点击左侧“工作”分组，看到所有工作相关软件（VS Code、Git Bash、Chrome）。他双击 VS Code，面板自动最小化。接着他再次呼出面板，这次在搜索框输入“需求”，项目网格立即显示“项目需求文档.docx”，双击打开，面板再次最小化。

**场景二：临时文件管理**
> 小王从 U 盘复制了一份报告到桌面。他不想让桌面变乱，于是把报告拖入 LaunchHub 面板，并给它打上“急需”标签。之后他只需呼出面板，点击“急需”标签，就能立即找到这份报告，无需在桌面翻找。

**场景三：跨分类检索**
> 小王的“微信”同时属于“聊天”和“常用”两个标签。他同时选中这两个标签（交集模式），看到的项目是同时标记了“聊天”和“常用”的（只有微信）。他切换为并集模式，看到的是所有标记了“聊天”或“常用”的项目。

---

### 3.2 功能需求（最终确定版）

| 编号 | 功能名称 | 优先级 | 详细描述 |
|------|---------|--------|---------|
| F1 | 初始化桌面导入 | 高 | 首次运行时，自动扫描桌面所有项目（.lnk、文件、文件夹），导入到“未分组”中。 |
| F2 | 手动添加项目 | 高 | 支持从资源管理器拖入文件/文件夹/.lnk 到面板，或通过按钮浏览添加。 |
| F3 | 分组管理 | 高 | 创建、重命名、删除自定义分组；分组互斥，一个项目只属于一个分组。 |
| F4 | 分组视图切换 | 高 | 左侧显示分组列表（含“全部”入口），点击切换右侧项目视图。 |
| F5 | 标签管理 | 高 | 创建、重命名、删除自定义标签。 |
| F6 | 标签归属 | 高 | 为一个项目添加/移除多个标签（通过右键菜单）。 |
| F7 | 标签筛选 | 高 | 顶部标签栏支持多选；默认交集模式，可切换为并集模式。 |
| F8 | 搜索过滤 | 高 | 顶部搜索框，输入关键词实时按项目名称筛选。 |
| F9 | 启动与打开 | 高 | 点击项目：.lnk 启动目标程序，文件/文件夹调用系统默认程序打开。 |
| F10 | 启动后最小化 | 高 | 启动/打开任意项目后，面板自动最小化到系统托盘。 |
| F11 | 右键上下文菜单 | 中 | 右键项目：编辑标签、移动到分组、从清单移除（不删除源文件）、打开所在文件夹。 |
| F12 | 路径有效性检测 | 中 | 启动时检查所有项目路径，失效项目显示灰色并提示“路径无效”。 |
| F13 | 全局热键 | 高 | 注册全局快捷键（默认 Alt+空格），呼出/隐藏面板。多屏环境下在鼠标所在屏幕弹出。 |
| F14 | 数据持久化 | 高 | 所有项目、分组、标签、设置保存到本地 JSON 文件，位于应用根目录（绿色版）。 |

### 3.3 非功能需求

| 类别 | 需求 |
|------|------|
| **性能** | 首次扫描桌面 < 1 秒；搜索过滤响应 < 100ms；启动面板动画流畅。 |
| **可靠性** | JSON 数据文件损坏时自动从备份恢复或重建默认结构，不崩溃。 |
| **可用性** | 呼出→搜索→启动，核心流程不超过 3 步。 |
| **安全性** | 不联网、不上传数据、不访问非用户指定的目录、不自动删除源文件。 |
| **可维护性** | 代码模块化，主进程与渲染进程职责分离；核心逻辑与 UI 解耦。 |
| **界面** | 干净、简洁、朴素、无视觉干扰；采用浅色主题。 |
| **可移植性** | 绿色版，所有数据与应用同目录，可直接拷贝使用。 |

### 3.4 核心概念定义

系统使用两个独立的组织维度：

| 维度 | 名称 | 特性 | 用途 |
|------|------|------|------|
| **分组** | Group | 互斥，一个项目只能属于一个分组 | 大类划分，每个分组有独立视图 |
| **标签** | Tag | 非互斥，一个项目可以有多个标签 | 跨分组的属性标记，用于筛选 |

**默认分组：** 系统自动创建“未分组”，所有新导入项目默认归属此分组。此分组不可删除。

### 3.5 明确排除的内容

-   不自动同步桌面新增项目（首次导入后由用户手动管理）。
-   不提供自动分类功能（用户手动打标签和分组）。
-   不提供多层嵌套文件夹或标签层级。
-   不提供皮肤/主题切换（初版仅浅色主题）。
-   不管理网络位置或已断开的移动设备项目（仅显示为失效）。

# 第四部分：总体设计

---

### 4.1 系统架构

采用 **Electron 主进程 + 渲染进程** 经典架构，模块化分层：

```
┌──────────────────────────────────┐
│         渲染进程（UI层）          │
│  - 主面板布局                    │
│  - 分组列表（侧边栏）            │
│  - 标签筛选栏                    │
│  - 搜索框                        │
│  - 项目网格视图                  │
│  - 右键上下文菜单                │
│  - 拖拽接收                      │
└──────────────┬───────────────────┘
               │ IPC 通信（15条通道）
┌──────────────▼───────────────────┐
│       主进程（服务层）            │
│  - 桌面扫描模块                  │
│  - 项目管理模块                  │
│  - 分组管理模块                  │
│  - 标签管理模块                  │
│  - 启动器模块                    │
│  - 数据持久化模块                │
│  - 全局热键模块                  │
│  - 系统托盘模块                  │
└──────────────────────────────────┘
```

### 4.2 模块划分

| 模块 | 进程 | 职责 |
|------|------|------|
| DataStore | 主 | 读写 data.json，数据备份与恢复 |
| DesktopScanner | 主 | 扫描桌面目录，返回文件列表 |
| ProjectManager | 主 | 项目的增删改查，路径有效性检查 |
| GroupManager | 主 | 分组的增删改 |
| TagManager | 主 | 标签的增删改，项目-标签关联管理 |
| LauncherStarter | 主 | 调用系统 Shell 启动程序/打开文件 |
| HotkeyManager | 主 | 注册/注销全局热键 |
| TrayManager | 主 | 系统托盘图标与菜单 |
| IpcHandlers | 主 | 注册所有 IPC 通道处理函数 |
| UI Components | 渲染 | 12 个 UI 组件（详见组件树） |

### 4.3 界面布局设计

```
┌─────────────────────────────────────────┐
│  [🔍 搜索项目名称...                    ] │  ← SearchBar
├──────────┬──────────────────────────────┤
│          │  标签筛选: [常用] [急需] [慢用] │  ← TagFilterBar
│  全部    ├──────────────────────────────┤
│  工作    │                              │
│  娱乐    │    [项目图标网格视图]          │  ← ProjectGrid
│  工具    │                              │
│  设计    │                              │
│          │                              │
│ [+新建]  │                              │
└──────────┴──────────────────────────────┘
```

**窗口规格：**
-   形态：浮动面板，无系统标题栏，自定义最小化/关闭按钮
-   默认尺寸：800 × 550 像素
-   弹出位置：鼠标所在屏幕中央
-   拖动：通过顶部搜索栏区域拖动窗口
